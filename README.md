# Предметная область: Расширенная система управления «Хогвартс»

## 1. Описание предметной области

В качестве предметной области выбрана информационная система школы чародейства и волшебства «Хогвартс». Система
предназначена для комплексного учета людей, академической успеваемости, спортивной жизни и системы безопасности.

## 2. Функциональные требования

1. **Отсутствие дублирования личных данных:** Система должна позволять человеку менять свой статус, при этом его базовые
   данные не должны вводиться заново.
2. **История баллов факультетов:** Система должна хранить не просто "текущий счет" факультета, а детальную историю
   транзакций: кто из сотрудников начислил/снял баллы, какому студенту, по какой причине и в какое точное время.
3. **Трекинг успеваемости:** Необходимо фиксировать, какие конкретно заклинания успешно освоил студент и в какую дату.
4. **Учет инцидентов:** Система должна логировать атаки и происшествия. В одном инциденте может участвовать несколько
   человек в разных ролях. Инцидент может быть инициирован конкретным злодеем.
5. **Турнир по Квиддичу:** Система должна хранить составы команд по позициям и фиксировать результаты матчей.

## 3. Бизнес-правила

1. Студент распределяется строго на один факультет на все время обучения.
2. Факультет может иметь только одну активную команду по Квиддичу.
3. Волшебная палочка в текущий момент времени принадлежит только одному владельцу, но владелец может поменяться.
4. В матче по Квиддичу всегда участвуют строго две команды.
5. Любое дисциплинарное взыскание или поощрение (изменение баллов) может быть осуществлено только сотрудником школы, но
   не студентом.

### Схема базы данных (Логическое проектирование)

База данных спроектирована в соответствии с требованиями 3НФ.

* **Количество таблиц:** 16
* **Файл схемы в формате DBML:** [schema.dbml](./schema.dbml)
* **Именование объектов:** Соответствует конвенциям PostgreSQL (snake_case, множественное число для таблиц, явное
  указание внешних ключей).

## 2. Развертывание и миграции

Инфраструктура разворачивается локально или на сервере с помощью Docker Compose.

### Быстрый старт

1. Скопируйте шаблон переменных окружения:
   ```bash
   cp .env.example .env
   ```

* Заполните необходимые параметры в .env.
* Запустите инфраструктуру:

```code
docker-compose up -d
```

* Управление миграциями и тестирование (Seqwall)
  Для управления схемой используется инструмент миграций (в формате .sql). Каждая миграция является идемпотентной и
  содержит rollback.
  В контейнере миграций настроена утилита Seqwall для тестирования по принципу staircase:
  Накат одной миграции: scripts/ci-up-one.sh
  Откат одной миграции: scripts/ci-down-one.sh
  При запуске контейнера сначала отрабатывает тестирование Seqwall на отдельной тестовой базе данных. При успешном
  прохождении тестов миграции применяются к основной БД до версии, указанной в MIGRATION_VERSION.
* Сидирование данных
  Генерация синтетических данных выполняется скриптом сидирования при условии `APP_ENV != 'prod'`.
  Объем генерируемых данных масштабируется пропорционально значению переменной `SEED_COUNT`.

## 3. Мониторинг

* Для наблюдения за состоянием инфраструктуры развернут стек Prometheus + Grafana:
* Создан выделенный пользователь PostgreSQL с ограниченными правами для сбора метрик.
* Метрики СУБД собираются с помощью postgres_exporter.
* Метрики объектного хранилища MinIO собираются напрямую встроенным функционалом MinIO.

## 4. Резервное копирование

* Резервное копирование выполняется по расписанию и загружается в хранилище MinIO:
* Периодичность: Задается через переменную окружения BACKUP_INTERVAL в формате cron.
* Глубина хранения: Настраивается через BACKUP_RETENTION_COUNT.
* В Grafana выводятся метрики размера последнего бэкапа и времени его создания.

## 5. Высокая доступность (HA Кластер)

Для отказоустойчивости СУБД развернута следующая конфигурация:

* Patroni: Управляет узлами PostgreSQL, координируя репликацию.
* etcd: Распределенный распределитель конфигурации (DCS) для выбора лидера.
* HAProxy: Единая точка входа для клиентов. Маршрутизирует запросы на запись к текущему мастеру на основе health checks.

## 6. Оптимизация производительности

Для проведения сравнительного анализа производительности до и после оптимизации используется инструмент **pgcompare**.

* Конфигурационный файл: pgcompare.yaml
* Запросы ДО оптимизации: queries_before.sql.
* Запросы ПОСЛЕ оптимизации: queries_after.sql.
* Результаты: Сгенерирован подробный отчет report.html, демонстрирующий изменение показателей `p95`, `p99`, `QPS` и
  планов выполнения

## 7. BI-метрики и CDC-пайплайн

Для бизнес-аналитики без нагрузки на транзакционную СУБД реализован Change Data Capture (CDC) пайплайн:

```text
[PostgreSQL] (Primary)
│ (Logical Replication - pgoutput)
▼
[Debezium (Kafka Connect)]
│
▼
[Kafka (KRaft mode)]
│
▼
[ClickHouse (Kafka Engine -> Materialized View -> MergeTree)]
│
▼
[Metabase Dashboard]
```

Реализованные шаги:

* Выбранная таблица: house_point_logs.
* Debezium: Настроен коннектор для отслеживания изменений и начальной выгрузки данных (snapshot.mode = initial).
  Настройка
  выполняется скриптом регистрации коннектора.
* ClickHouse: Применен паттерн интеграции через движок Kafka, данные трансформируются и сохраняются в таблицы семейства
  MergeTree с помощью Materialized View.
* Визуализация: В Metabase настроен аналитический дашборд, отображающий ключевые метрики.
